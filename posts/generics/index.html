    The history of generics in go is trully amazing. The ultimate answer to “generics question” in go has evolved together with the language.

    Originally, go official FAQ answered it in the folling way.

    Why does Go not have generic types?
Generics may well be added at some point. We don't feel an urgency for them, although we understand some programmers do.
Generics are convenient but they come at a cost in complexity in the type system and run-time.
We haven't yet found a design that gives value proportionate to the complexity, although we continue to think about it.
Meanwhile, Go's built-in maps and slices, plus the ability to use the empty interface to construct containers (with explicit unboxing)
mean in many cases it is possible to write code that does what generics would enable, if less smoothly.
This remains an open issue.

    But the answered has recently changed and now FAQ we have there.

    A language proposal implementing a form of generic types has been accepted for inclusion in the language. If all goes well it will be available in the Go 1.18 release.
Go was intended as a language for writing server programs that would be easy to maintain over time. (See this article for more background.)
The design concentrated on things like scalability, readability, and concurrency.
Polymorphic programming did not seem essential to the language's goals at the time, and so was left out for simplicity.
The language is more mature now, and there is scope to consider some form of generic programming. However, there remain some caveats.
Generics are convenient but they come at a cost in complexity in the type system and run-time.
We haven't yet found a design that gives value proportionate to the complexity, although we continue to think about it.
Meanwhile, Go's built-in maps and slices, plus the ability to use the empty interface to construct containers (with explicit unboxing) mean in many cases it is possible to write code that does what generics would enable, if less smoothly.
The topic remains open. For a look at several previous unsuccessful attempts to design a good generics solution for Go, see this proposal.

    Let’s take a deeper look how we got there. It all started since day 0 of go public appearance in 2010 with, in my opinin, the smart choise to not include generics in the intial language release. Soon enough go conquered some popularity and developed community around. Along with this, came endless complaints about lack of generics in go and related comparisons with Java and other languages alike. I believe, this can be explained by the developers migration from other languages and their desire to make go look like mainstream language x.

    Let’s also analyze what people wanted exactly and what asked for. Go is awesome, it has yearly official survey ritual to gather comminity opinion on various vital questions. Sad part though, there was not public go survey until year of 2016. Though, probably, it’s not that hard to find hundreds of articles with outcry complaining about lack of generics in go, daiting all the way back to the go release date. Still we will use this public surveys results as the source of the analyze. Once again, it’s a bit “fresh” and doesn’t fully represent full historical perspective of generics barganing, however it verifies the fact that genercis desire didn’t vanished within passing years. 2016 - generics are the first on What changes would improve Go most?

    The problems with generics, as I see them

    #1 They make existing go type system unsound.

    Go has “limited generics” inside already (kind of) - in form of maps and slices. Introduing new form of user defined genrics that is ortoghonal to those type will make existing map and slice to not fit. Soon enough user defined vectors and maps with (better performance, better interface, better whatever) will follow, which inenventeably lead to some code base use internal containers everywhere (really knonw problem in C++).

    2 Their use case is really narrow, but they add a lot of weight to the language.#

    Well, all knows why generics are useful - generics alghoritms and containers. The thing with go it doesn’t need neither of those that desperately. Most of the go projects are designed to be small and to use service based approach. Sure those smaller projects could benefit from generics too and they already do by using maps and slices. But they DO NOT need user defined generics most of the times, it only will bring complexity and bloat to the their code, in my opinion. Also looking back on https://blog.golang.org/survey2020-results #paste a screenshot here p90 for all applications written in go is either server side API/RPC code or data processing pipelines or some tooling around this. For all of these domains, I personally, find generics rarely useful. But what about those bigger projects _kubernetes _coacroach _etc., if you ask me on this, I’ll answer golang shouldn’t even try match their needs. Because in my opinion, it’s impossible to match n radically different requirements for n projects which size is comparable to go itself.

    3 They are impossible to take back.#

    Generics is a huge change, maybe the biggest change that will be ever introduced to go (from cognitive perspective). And the core team desire to add generics to MINOR 1.18 version looks strange to me whatever angle I look at it. I mean, yes it “technically meets semver specification” of “version when you add functionality in a backwards compatible manner”, but common. I’d expect it to be MAJOR go2 release with huge implications:

    • for new libraries development
    • rewriting existing user code
    • rewriting std library code
    • etc.

    4 They left secod half of comunity dissapointed.#

    After years and years of go community defeneding absense of generics and learning to live with tradeoff of simplicity to features. We suddenly got 180 degree turn in 1 year. Changing people mentality and believes is not a simple process, and I’d imagine a lot of people got frustrated becuase of this turn. Subjectively, despite of surveys results, I think, that true majority of people working with go on daily basis didn’t want generics.

    5 Generic via boxing doesn’t give anything valuable to go.#
